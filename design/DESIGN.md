# 总体设计原则

- 异步优于同步。
- 事件驱动优于轮询。
- 事件需要被正确消费和响应，不允许偶然竞争下的丢失或被遗忘。
  - 想象你在设计一个计算机操作系统级别的事件系统，事件必须被正确处理，否则系统就会崩溃、或者用户会觉得系统不可靠。
- 保持响应，要么响应用户，要么告诉用户自己在具体忙于什么。
- 保持简洁，符合 `zig zen`。

# Agent Loop OS 设计

## 原则

- 本身完全事件/消息队列驱动，不要和某一种前端（例如 TUI、WebUI）的交互形式绑定。
- 队列可以存在抢占式（例如用户的查询）或非抢占式（例如定时任务）的不同优先级任务。
- 输出以事件流的方式传出。
  - 粒度越细越好，保证事件发生至抵达用户的时间尽可能短。
  - 由于消息队列的排队属性，当开始处理一个事件时，也要提供开始的信号，以让用户知道哪些请求正在被处理、哪些还在排队。
- 可能同时运行多个 Agent 实例（例如 Multi-Agents，相互之间不共享状态），也可能同时调用多个工具，因此要维护好这些执行状态，做到可打断。
- 和操作系统相比，类比起来就是：
  - 工作记忆 = 进程虚拟内存
  - 永久记忆 = 磁盘文件系统
  - 事件系统 = 外部中断输入、时钟中断等
  - Agent = 进程
  - 工具调用 = 系统调用 / 内核 Worker
  - 主 Agent = 桌面环境 / init

## 架构

- UI:
  - 一个最终的消息监听接口，Agent Loop OS 的以下各层会通过这个接口发送事件。外部的 TUI、WebUI 等等都可以通过这个接口来接收事件并展示给用户。
- Userspace：
  - Agent：相当于进程。维护自己的工具记忆（简化到 LLM 中就是 message history）、调用 LLM 接口。
    - Syscall：当 Agent 需要调用工具时，发出一个系统调用事件，进入 Kernel。
    - 信号：一个输入命令队列。当 Agent 需要响应外部事件（例如用户输入、工具调用完成等）时，Kernel 会通过信号的方式通知 Agent。
- Kernel：
  - Runtime：相当于操作系统内核循环，抽象地来看，只做三件事：处理中断（将中断分发给对应的 Agent）、维护 Agent 列表、维护 Tool Worker 列表。
    - 外部中断：来自用户的查询、回复、取消等请求。
    - 内部中断：来自异步工具调用的完成。
  - Tool Worker：相当于内核线程，专门用来执行工具调用。工具调用的结果通过内部中断或系统调用的方式回到 Runtime。
    - 同步系统调用（如 ask_user）直接阻塞调用，立刻返回结果。
    - 异步系统调用（如 code_run）在 Syscall 侧立刻返回结果。稍后在 Worker 中执行完成时，触发一个内部中断事件，这个事件又会通过信号的方式通知 Agent。

## 打断设计

- 任何持续中的任务都应该可打断（Cancelable）。目前的任务至少包括：
  - Agent：正在处理的 LLM 调用（包括等待流式响应的过程）。
    - 注意：如果在流式响应过程中被打断了，应该立刻停止处理这个响应，返回一个表示未完成消息但已结束的事件（例如 `message_incomplete`），而不是直接丢弃这个消息。这样用户就知道这个消息没有完成，而不是完全没有反应。
    - 当要打断一个 agent 时，其实是要打断所有接下来可能触发继续回复的事件，例如工具调用完成的事件、用户新消息的事件等等。也就是说，打断一个 Agent 就是要让它尽快进入一个 idle 状态，不再处理任何事件（包括下面说的 tool_cancelled 事件），直到用户或其他外部中断再次发出指令让它继续工作。
  - Tool Worker：正在执行的工具调用。
    - 注意：如果工具调用被打断了，应该立刻停止工具的执行，返回一个表示工具调用被取消的事件（例如 `tool_cancelled`）。
- 可能的做法是分配每个持续任务一个 Cancel Context，在每个 Agent 中维护一个 [cancel context] 的列表。当需要打断一个 Agent 时，Agent 收到 Cancel 信号后就找到对应的所有相关 cancel context，触发它们的取消信号。

# 记忆设计

## 原则

- 记忆始终存在，永远不会重置（重装应用或为了调试、临时解决 bug 和开发目的而清除记忆除外）。
