# 编码风格 / 重构指导

- 锁、并发、异步相关的代码要格外注意，保持清晰，清楚地说明每个锁的目的、范围和使用方式。
- clone、free 等资源管理一定要放在对应的结构体中，不要散落在外面。
- 不要添加无意义的包装函数，尤其是那些只是简单调用另一个函数的函数，或者只被调用一次的函数。
- 将相似的逻辑聚合在附近，保持空间局部性，避免过度分散。
- 警惕「巨型状态结构体」。如果一个结构体包含了过多的字段，考虑将其拆分成更小、更专注的结构体，或者严格地考虑每个字段的必要性，添加注释说明为什么需要它们，将相关的字段聚合在一起。
- 同样地，警惕包含太多函数的「巨型模块/结构体」。
- 对于每一个功能项，选择适合的数据结构和算法抽象，保持代码的清晰性。
- 如果一个函数充斥的没有逻辑的代码（如大量的 if-else 嵌套、超长的判断链、访问几十个字段的代码），说明你的设计有问题，函数应该更专注于一个单一的功能，或者你需要引入新的抽象来简化代码。
- 注意内存释放、Cancellation 资源管理等有责任的代码逻辑。一般的范式是「谁申请，谁释放」「谁拥有，谁销毁」，保持资源的生命周期清晰，除非涉及性能问题，否则不要传递资源的所有权，或者在多个地方释放同一个资源。尤其是 Cancellation，应该思考如何将 cancel 的信号清晰地传递给拥有需要 cancel 资源的链路（通常是自己的 caller），而不是在一个地方集中处理所有的 cancel/free 逻辑。比如说 A->B->C，如果 C 中被 cancel 了，不要编写一个什么 handleCancelC 的方法然后在里面一口气释放 A、B、C 的资源，而是应该 C 释放自己的资源，返回一个 cancel 的信号给 B，B 释放自己的资源，返回一个 cancel 的信号给 A，A 释放自己的资源。这样每个模块只需要关注自己资源的管理，减少了耦合和复杂度。
- 重构中不要随便拆分文件/结构体。拆分不降低复杂度，反而可能增加复杂度，需要在多个实体之间跳转才能理解代码。在一个文件中提高代码的清晰度和可维护性，而不是通过拆分文件来掩盖复杂性。
- 清晰追踪 allocator 和 io 的使用。一般来说，一个数据体总应该被同一个 allocator 管理（即便是同一个 allocator 到处传也不好。只要传递到了另一个地方，在这个函数/模块内就应当该被视为一个新的 allocator）；一个 task 的生命周期应该被同一个 io 管理（同样的道理）。
